# Copilot report

This is a source-and-docs read-through, not a runtime exploit writeup. The biggest theme is **bootstrap drift**: the
in-process guards are materially stronger than the generated `sitecustomize` path, but several docs currently describe
them as equivalent.

## Additional security-sensitive Python features worth considering as opt-in guards

| Candidate feature                                                                                                       | Why it matters                                                                                                                                                                                                |
|-------------------------------------------------------------------------------------------------------------------------|---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| **Environment guard** (`os.environ`, `os.getenv`, `os.putenv`, `os.unsetenv`)                                           | Secrets often live in env vars. Today a blocked program can still read API keys from the environment and, if subprocesses are allowed, rewrite child-process behavior.                                        |
| **Dynamic code execution guard** (`eval`, `exec`, `compile`, `runpy.run_*`)                                             | This would make LLM/plugin abuse noisier and harder by blocking runtime code generation / secondary execution paths.                                                                                          |
| **Import-policy guard** (generic deny/allow list beyond native modules)                                                 | The current import guard is specialized for native/FFI. A general `--deny-import pickle --deny-import marshal --deny-import webbrowser ...` feature would be high leverage and reuse the same basic approach. |
| **Interpreter mutation guard** (`sys.path` edits, `site.addsitedir`, `os.chdir`, selected `importlib` mutation helpers) | These APIs let code reshape where imports resolve from, which weakens assumptions around hermetic startup order and filesystem confinement.                                                                   |
| **Introspection-hardening profile** (`gc`, `inspect`, `sys._getframe`, maybe `importlib.reload`)                        | This would not be bulletproof, but it would make `--seal` more meaningful against "mildly mischievous" code by removing the easiest routes to recover originals and patch around the guards.                  |

## Likely root cause

`scripts\build_bootstrap.py` says it "avoids code duplication and ensures that the bootstrap logic is always in sync",
but each guard still carries a hand-maintained `BOOTSTRAP_CODE` string snapshot. So the script only assembles those
snapshots; it does not derive bootstrap behavior from the live guard implementation. That explains the current parity
drift.

**Evidence:** `scripts\build_bootstrap.py:3-12`, `scripts\build_bootstrap.py:103-116`, plus each guard module's
`BOOTSTRAP_CODE` block.
