# Code Review: Hermetic

**Date:** April 28, 2026
**Reviewer:** Gemini CLI

## Overview

Hermetic is a user-space sandbox runner for Python that uses monkey-patching to restrict potentially dangerous APIs (
network, subprocess, filesystem writes, etc.). It acknowledges its own limitations as a "fig leaf" of security, which is
an honest and accurate assessment.

## High-Level Findings

### 1. Architectural Integrity & Bypasses

The project correctly identifies that monkey-patching is easily defeated by native code, direct C-level calls, or import
order tricks.

- **Flaw**: The `_socket` module is deliberately not patched (documented), which is a major egress route for any
  sophisticated attacker.
- **Flaw**: `ExtensionFileLoader` is patched, but `_ctypes` and `_cffi_backend` are only blocked if the user hasn't
  already loaded them or if they use standard import paths.

### 2. Configuration & State Management

There is significant confusion between `BlockConfig` (in `blocker.py`) and `GuardConfig` (in `profiles.py`).

- **Usability Challenge**: `BlockConfig` uses names like `block_network`, while `GuardConfig` uses `no_network`. This
  inconsistency makes the codebase harder to maintain and reason about.
- **Risk**: Merging logic for these configs is duplicated and slightly different.

### 3. Resource Management

- **Bug (Leak)**: `hermetic.bootstrap.write_sitecustomize` creates a temporary directory using `tempfile.mkdtemp` but
  never cleans it up. Repeated use of `bootstrap` mode (e.g., running many child processes) will clutter the system's
  temp directory.

### 4. Filesystem Guard Gaps

The "readonly" filesystem guard is primarily focused on `open` and mutation functions like `os.remove` or
`pathlib.Path.unlink`.

- **Flaw**: Enumeration functions such as `os.listdir`, `os.walk`, `os.scandir`, and metadata functions like `os.stat`
  are **not** blocked. An attacker can map out the entire filesystem even when restricted to a `fs_root`.
- **Missing Feature**: There is no "block all reads" mode; `fs_readonly` only restricts writes and *optionally*
  constrains reads to a root.

### 5. Cross-Platform & Windows Issues

- **Usability/Bug**: In `hermetic.resolver.resolve`, the detection of the correct interpreter for Windows console
  scripts (`.exe` wrappers) likely fails because they are PE binaries, not text files with shebangs. This might cause
  `hermetic` to incorrectly choose `inprocess` mode for scripts that belong to a different virtual environment.
- **Windows UX**: On Windows, `os.execve` is simulated via `subprocess.run` followed by `sys.exit`. This is a reasonable
  workaround but doesn't handle signals or true process replacement as seamlessly as Unix.

### 6. Development Workflow & Maintainability

- **Maintenance Burden**: `hermetic/bootstrap.py` is auto-generated by a manual script (`scripts/build_bootstrap.py`).
  If a developer modifies a guard module but forgets to run this script, `bootstrap` mode will use stale logic. This
  should be automated (e.g., via a pre-commit hook or build step).
- **DRY Violation**: Each guard module contains a `BOOTSTRAP_CODE` string literal that is a copy of its logic. This
  leads to double-maintenance.

## Detailed Observations

### Dead Code & Stale Data

- **Dead Code**: `dead_code/draft.py` is an old version (calling itself "envelope") and should be removed.
- **Version Inconsistency**: `hermetic/__init__.py` reports version `0.1.0`, while `pyproject.toml` and
  `hermetic/__about__.py` report `0.2.0`.
- **Hardcoding**: `_METADATA_HOSTS` in `network.py` is a hardcoded list of IPs. While common, it might miss some niche
  cloud providers.

### Missing Features

- **Block `sys.modules` mutation**: An attacker can replace `socket` in `sys.modules` with a fresh copy or a mock that
  they control.
- **Environment Block**: `os.environ` is replaced with a `_GuardedEnviron`, but `os.environb` (bytes version) might
  still be accessible if not explicitly handled (it is handled in bootstrap but check if fully covered in standard
  install).

## Recommendations

1. **Unify Config**: Consolidate `BlockConfig` and `GuardConfig` into a single class with consistent naming.
2. **Fix Temp Leak**: Use a context manager or a cleanup routine in `runner.py` to delete the temporary `sitecustomize`
   directory after the child process exits.
3. **Harden Filesystem Guard**: Block directory enumeration and metadata access when a `fs_root` is specified.
4. **Automate Bootstrap Generation**: Add a check in CI or a pre-commit hook to ensure `hermetic/bootstrap.py` is in
   sync with the guard modules.
5. **Remove Dead Code**: Delete `dead_code/` directory.
6. **Fix Versioning**: Synchronize version numbers across all files.
7. **Improve Windows Resolution**: Research a way to detect the interpreter for Windows `.exe` console scripts (perhaps
   by checking the directory structure or using `pip`'s internal knowledge).
